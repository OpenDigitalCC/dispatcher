## Pairing: pre-flight writability check

**Why:** If `dispatcher-agent request-pairing` is run without sudo, it
successfully connects to the dispatcher and queues a pairing request, but then
fails to write the received certs to `/etc/dispatcher-agent/`. The operator
sees a confusing error after the connection has already been made, and a stale
pairing request is left in the dispatcher queue.

**Fix:** In `bin/dispatcher-agent`, in the `request-pairing` mode handler,
before calling `generate_key_and_csr` or making any network connection, check
that the cert directory is writable by the current user. Die immediately with a
clear message if not.

**Where:** `bin/dispatcher-agent` - `mode_request_pairing` function (or
equivalent). The cert directory comes from the config: `dirname($config->{cert})`.

```perl
use File::Basename qw(dirname);
my $cert_dir = dirname($config->{cert});
unless (-w $cert_dir) {
    die "Cannot write to $cert_dir - re-run with sudo\n";
}
```


## Pairing: stale request expiry

**Why:** Failed pairing attempts (e.g. run without sudo as above) leave `.json`
files in `/var/lib/dispatcher/pairing/`. These accumulate and clutter
`list-requests`. The polling window is 10 minutes, so any request older than
that will never be collected by a waiting agent.

**Fix:** Add a `_expire_stale_requests` private function to
`Dispatcher::Pairing` that deletes `.json` files (without a corresponding
`.approved` or `.denied`) older than 10 minutes. Call it at the start of
`run_pairing_mode` and `list_requests`.

**Where:** `lib/Dispatcher/Pairing.pm`

```perl
sub _expire_stale_requests {
    my ($pairing_dir) = @_;
    my $cutoff = time() - 600;   # 10 minutes
    opendir my $dh, $pairing_dir or return;
    while (my $f = readdir $dh) {
        next unless $f =~ /^(.+)\.json$/;
        my $base = $1;
        my $path = "$pairing_dir/$f";
        next if -f "$pairing_dir/$base.approved";
        next if -f "$pairing_dir/$base.denied";
        unlink $path if (stat $path)[9] < $cutoff;
    }
    closedir $dh;
}
```


## Pairing: request nonce

**Why:** The agent currently sends only `hostname` and `csr` in the pairing
request. If the same agent re-pairs (or a failed attempt is retried), there is
no way to distinguish responses. A nonce generated by the agent makes each
request unique and allows the agent to verify the response is for its specific
request, not a replayed or misrouted one.

**Fix:** Agent generates a random hex nonce alongside the CSR. It is included
in the JSON pairing request, stored in the dispatcher's `.json` file, and
echoed in the `.approved` response. Agent verifies nonce on receipt before
storing certs.

**Wire format change:**

Pairing request gains `nonce` field:
```json
{ "hostname": "sjm-explore", "csr": "...", "nonce": "a3f9b2c1d4e5f607" }
```

Approval response gains `nonce` field:
```json
{ "status": "approved", "cert": "...", "ca": "...", "nonce": "a3f9b2c1d4e5f607" }
```

**Where:**
- `lib/Dispatcher/Agent/Pairing.pm` - `generate_key_and_csr` or caller: generate
  nonce with `sprintf '%016x%016x', int(rand(0xffffffffffffffff)), $$`
- `lib/Dispatcher/Agent/Pairing.pm` - `request_pairing`: include nonce in
  payload, verify nonce in response before calling `store_certs`
- `lib/Dispatcher/Pairing.pm` - `_handle_pair_request`: extract and store nonce
  in `.json` file
- `lib/Dispatcher/Pairing.pm` - `approve_request`: echo nonce in `.approved` response


## Reqid entropy

**Why:** `gen_reqid()` in `Dispatcher::Engine` uses `int(rand(0xffffffff))` with
Perl's default seeding. Across process restarts the sequence can repeat, producing
duplicate reqids in logs and making cross-host log correlation unreliable.

**Fix:** Replace with a generator combining `Time::HiRes::time()`, PID, and a
per-process counter.

**Where:** `lib/Dispatcher/Engine.pm` - `gen_reqid()` function.

```perl
my $_reqid_counter = 0;

sub gen_reqid {
    require Time::HiRes;
    my $t   = int(Time::HiRes::time() * 1000) & 0xffff;
    my $pid = $$ & 0xffff;
    my $seq = (++$_reqid_counter) & 0xffff;
    return sprintf '%04x%04x%04x%02x',
        $t, $pid, $seq, int(rand(0xff));
}
```

Also applies to the pairing nonce (see above) and `_gen_reqid` in
`Dispatcher::Pairing`.


## Interactive pairing mode

**Why:** Currently `pairing-mode` blocks on `accept()` and the operator must
open a second terminal to run `list-requests` and `approve`. A single-terminal
interactive flow is more practical.

**Approach:** Use `IO::Select` to multiplex the server socket and STDIN. When a
request arrives, print it immediately and prompt. If a second request arrives
before the operator responds, display the updated queue and re-prompt.

**Prompt on single request arrival:**

```
Pairing request from sjm-explore (192.168.125.125) - ID: 1a15334d
  Received: 2026-03-05T16:32:01Z
Accept, Deny, or Skip (to decide later)? [a/d/s]:
```

**Queue display when multiple pending:**

```
2 pending pairing requests:
  [1] sjm-explore    192.168.125.125   2026-03-05T16:32:01Z
  [2] prod-db-01     192.168.125.200   2026-03-05T16:32:45Z
Command (a1/d1/a2/d2/list/quit):
```

**Where:** `lib/Dispatcher/Pairing.pm` - `run_pairing_mode`. Significant
rework: the current `while { $server->accept }` loop becomes an `IO::Select`
select loop. The select set contains the server socket; STDIN is added when
interactive (i.e. when STDIN is a tty: `-t STDIN`). Falls back to current
non-interactive behaviour if STDIN is not a tty (e.g. when run from a script).

`bin/dispatcher` - `mode_pairing`: no changes needed; options and log_fn pass
through as before.


## Agent tags and metadata

**Why:** Discovery currently returns scripts and version only. Operators need to
group and filter agents by environment, role, location, or other dimensions
without querying each agent individually.

**Approach:** Add a `[tags]` section to `agent.conf` (or a separate
`tags.conf`) with arbitrary key=value pairs. Tags are loaded at startup,
reported in `/capabilities`, included in `/discovery` output, and shown in
`list-agents`.

**Config format** (in `agent.conf` or a dedicated `tags.conf`):

```ini
[tags]
env  = prod
role = db
site = london
```

**Capabilities response gains `tags`:**

```json
{
  "status": "ok", "host": "sjm-explore", "version": "0.1",
  "tags": { "env": "prod", "role": "db", "site": "london" },
  "scripts": [...]
}
```

**Discovery response includes tags per host:**

```json
{
  "ok": true,
  "hosts": {
    "sjm-explore": {
      "status": "ok", "version": "0.1", "rtt": "68ms",
      "tags": { "env": "prod", "role": "db" },
      "scripts": [...]
    }
  }
}
```

**Where:**
- `lib/Dispatcher/Agent/Config.pm` - add `load_tags($path)` function, or extend
  `load_config` to parse a `[tags]` section
- `bin/dispatcher-agent` - load tags at startup, pass to `handle_capabilities`,
  reload on SIGHUP alongside allowlist
- `bin/dispatcher-agent` - `handle_capabilities`: include `tags` in response
- `lib/Dispatcher/Engine.pm` - `_capabilities_one`: no change needed; tags
  come through in the JSON response automatically
- `bin/dispatcher` - `mode_list_agents`: display tags column or append tag
  key=value pairs

**Questions to resolve before implementing:**
- Tags in `agent.conf` under a `[tags]` section, or a separate `tags.conf`?
  Separate file is cleaner for SIGHUP reload and permissions, but adds a file.
- Should `list-agents` show all tags inline, or just a count with detail on
  `show-agent <hostname>`?

## Installer: `sudo` prefix on suggested commands

**Why:** Post-install next-steps are shown to the operator who ran the installer
as root, but they may not remain root for subsequent steps. Commands like
`systemctl enable`, `cp`, `chmod` need sudo for non-root users.

**Where:** `install.sh` - `print_next_steps_agent`, `print_next_steps_dispatcher`,
`print_next_steps_api` functions. Prefix commands with `sudo` where root is
required.

**Why:** The current next-steps output lists four manual `openssl` commands to

**Approach:** Add a `generate_dispatcher_cert` function to `Dispatcher::CA` (or

**Where:**

- `bin/dispatcher` - add `setup-dispatcher` mode (or extend `setup-ca` to
  optionally generate the dispatcher cert in the same step)
- `install.sh` - update next-steps to reference `dispatcher setup-dispatcher`
  instead of listing manual openssl commands


## Cert renewal

**Why:** Agent certs expire after 825 days with no automated renewal path.
The current workaround is a full re-pair, which works but requires manual
operator intervention on both hosts.

**Approach:** A `renew` mode that re-uses the existing pairing flow but skips
key regeneration (reuses the existing agent key). The agent submits a new CSR
for its existing key; the dispatcher signs it; the agent stores the new cert.
No allowlist or config changes needed.

**Where:**
- `bin/dispatcher-agent` - add `renew` mode alongside `request-pairing`
- `lib/Dispatcher/Agent/Pairing.pm` - add `generate_csr_only(%opts)` that skips
  key generation and uses the existing `agent.key`
- `lib/Dispatcher/Pairing.pm` - no changes needed; renewal uses the same
  pairing flow
- `lib/Dispatcher/Registry.pm` - update `expiry` field in agent record after
  renewal completes


## Alpine Linux installer

**Why:** Dispatcher should support Alpine Linux for lightweight container and
edge deployments.

**Key differences from Debian installer:**

- Package manager: `apk add` instead of `apt install`
- Package names: verify availability in Alpine community repo:
  - `IO::Socket::SSL` → `perl-io-socket-ssl`
  - `JSON` → `perl-json`
  - `LWP::UserAgent` → `perl-lwp-useragent`
  - `perl-base` equivalents → `perl` (Alpine bundles differently)
- Service management: OpenRC instead of systemd
  - Service script in `/etc/init.d/dispatcher-agent` (shell script, not unit file)
  - `rc-update add dispatcher-agent default` instead of `systemctl enable`
  - `rc-service dispatcher-agent start` instead of `systemctl start`
- User creation: `adduser -S -D -H -s /sbin/nologin dispatcher-agent` (Alpine
  `adduser` syntax differs from Debian `useradd`)

**Deliverable:** `install-alpine.sh` following the same structure as `install.sh`
with role flags `--agent`, `--dispatcher`, `--api`, `--uninstall`. Shared logic
can be factored into a common `install-common.sh` if significant overlap exists.

**Note:** Verify all Perl module package names in Alpine community repo before
finalising. Some may require enabling the community repo explicitly.


## Red Hat / Rocky / Alma installer

**Why:** Dispatcher should support RHEL-family distributions for enterprise
deployments.

**Key differences from Debian installer:**

- Package manager: `dnf install` instead of `apt install`
- Package names:
  - `IO::Socket::SSL` → `perl-IO-Socket-SSL`
  - `JSON` → `perl-JSON`
  - `LWP::UserAgent` → `perl-LWP-UserAgent`
  - Core modules bundled in `perl-core` or `perl`
- SELinux: likely to block agent from binding non-standard ports (7443, 7444,
  7445) and executing scripts. Options:
  - `semanage port -a -t http_port_t -p tcp 7443` (and 7444, 7445)
  - Custom SELinux policy module (more robust, more complex)
  - Document manual steps if automatic policy changes are undesirable
- Firewall: `firewall-cmd --add-port=7443/tcp --permanent` may be needed on
  agent hosts; document as a post-install step rather than automating
- User creation: `useradd` syntax compatible with Debian version
- systemd: unit file format identical; no changes needed

**Deliverable:** `install-redhat.sh` following the same structure as `install.sh`.

**Note:** Test on Rocky Linux 9 as the primary target. Verify SELinux behaviour
before committing to an approach - the automatic `semanage` approach requires
`policycoreutils-python-utils` which may not be present.


## Unpairing

**Why:** There is currently no way to remove an agent from the registry or
revoke its trust. A compromised or decommissioned host remains in the registry
and its cert remains valid until expiry (825 days).

**Approach:**
- `dispatcher unpair <hostname>` CLI mode
- Removes `/var/lib/dispatcher/agents/{hostname}.json` from the registry
- The agent cert remains technically valid (no CRL support currently). Options:
  - Accept this limitation - the allowlist on the agent is the real gate
  - Implement a simple CRL: maintain `/etc/dispatcher/ca.crl`, have the agent
    check it on startup or via SIGHUP
  - Re-issue all certs with a new CA (nuclear option, requires re-pairing all agents)

**Where:**
- `lib/Dispatcher/Registry.pm` - add `remove_agent(%opts)` function
- `bin/dispatcher` - add `unpair <hostname>` mode
- `lib/Dispatcher/CA.pm` - optionally add `revoke_cert` and `generate_crl` if
  CRL support is in scope

**Recommended scope for initial implementation:** registry removal only, with a
clear warning that the cert remains valid until expiry and the agent should be
decommissioned promptly.


## Output module

**Why:** `_format_run_results()` and `_format_ping_results()` in `bin/dispatcher`
are private functions in the CLI script and cannot be unit tested without
invoking the full CLI. Formatting bugs are only caught by integration testing.

**Fix:** Extract to `lib/Dispatcher/Output.pm` as public functions. The CLI
calls them by module reference. Tests in `t/output.t` can test formatting
directly with synthetic result data.

**Where:**

  `format_ping_results(\@results)`, `format_agent_list(\@agents)`,
  `format_discovery(\%hosts)`
- `bin/dispatcher` - replace inline `_format_*` functions with
  `Dispatcher::Output::format_*` calls
- New file: `t/output.t`
